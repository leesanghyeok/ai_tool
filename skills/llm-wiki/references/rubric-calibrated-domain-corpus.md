# 루브릭 기준 도메인 corpus 구축 패턴

사용자가 “wiki 안의 내용만으로 답변”하면서 특정 루브릭 점수 기준을 통과하라고 요청할 때 쓰는 운영 메모.

## 핵심 원칙
- 스킬 본문, SCHEMA taxonomy, 일반 지식은 답변 근거가 아니다. 실제 wiki 본문과 raw에서 컴파일된 concept/comparison/query 페이지만 답변 근거로 쓴다.
- 공개 인터넷 자료는 도메인 원칙과 프레임워크를 보강할 수 있지만, 회사별 baseline·unit economics·경쟁자별 실제 수치가 필요한 루브릭 90점 조건을 자동으로 충족시키지는 못한다.
- 루브릭이 “회사별 데이터가 없으면 높은 점수를 주지 않는다”처럼 실제 데이터 의존 조건을 갖고 있으면, 허위 수치를 만들지 말고 블로커로 명시한다.

## 효율적인 작업 순서
1. 질문 세트를 읽고 문항을 도메인 클러스터로 나눈다.
2. 클러스터별로 서브에이전트를 병렬 투입해 URL 후보, 핵심 claim, 쓰이는 질문을 조사하게 한다. 서브에이전트는 파일을 쓰지 않게 한다.
3. 부모 세션에서 URL 후보를 검증한 뒤 `_meta/*source-manifest*.json`에 저장한다.
4. fetch 스크립트로 raw/articles/<cluster>/ 아래에 원문을 저장하고, `source_url`, `fetched_via`, `ingested`, `sha256`, `cluster`, `questions`, `title` frontmatter를 붙인다.
5. raw를 직접 답변하지 말고 concept/comparison/query 페이지로 컴파일한다.
6. index.md와 log.md를 업데이트한다.
7. 답변을 만들 때 각 문항별 근거 페이지를 `[[wikilink]]`로 명시한다.
8. 루브릭 hard-cap 조건을 자동 체크할 수 있으면 `_meta/validate_*.py` 같은 검증 스크립트로 1차 누락을 잡는다.
9. 의미 기반 엄격 리뷰를 별도 서브에이전트에 맡기고, 90점 미만 원인을 문항별로 기록한다.
10. 실제 데이터가 없어 종료조건을 달성할 수 없으면, “현재 corpus로 가능한 답변”과 “90점 달성에 필요한 실제 데이터 체크리스트”를 분리해 저장한다.

## 피해야 할 것
- “인터넷 자료를 모두 학습”이라는 표현을 문자 그대로 수행한다고 말하지 않는다. 범위를 “루브릭 통과에 필요한 최소 충분 corpus”로 운영한다.
- 루브릭 통과를 위해 존재하지 않는 회사 baseline, CAC, LTV, churn, MDE, 경쟁 수치를 만들어내지 않는다.
- SCHEMA의 태그 taxonomy만 보고 구체 전략을 추론하지 않는다.
- 자동 키워드 검증 통과를 90점 통과로 보고하지 않는다. 자동 검증은 hard-cap 누락 탐지일 뿐이다.
